mini-agent — Complete Reference Documentation


## 1. Project Overview

mini-agent is a state-machine-driven AI agent orchestrator built as a proof-of-concept without any agent framework. It does not use LangChain, LangGraph, AutoGen, or OpenAI's native tool-calling API. Every decision, transition, and tool invocation is driven by structured LLM output combined with a JSON-configured state graph. The result is a fully transparent and deterministic agent loop where every intermediate step is logged and inspectable.

The design demonstrates that a production-quality agent can be built from three simple ideas: a directed state machine that routes execution, an LLM that declares its intent as a structured Pydantic object, and a per-session memory store that accumulates all context across turns.

mini-agent supports two primary use cases. As a document helper it can answer questions from a local knowledge base of documents loaded at startup, combining retrieval-augmented generation with web search when the knowledge base does not cover the topic. As a project planner it accepts a high-level goal, breaks it into a structured research and execution plan, carries out each step using available tools, and delivers a comprehensive result — then stays in conversation for iterative refinement.


## 2. Architecture: The State Machine

The state machine is defined entirely in a JSON configuration file at custom/default/stateMachineConfig.json. Every state in the graph is a JSON object with three required fields: function (the name of the Python method to call), args (a list of arguments, where any string value matching the pattern dollar-sign-open-brace-variable-name-close-brace is substituted from StateMemory at runtime), and nextState (either a string naming the next state, or an array of conditional transition objects).

When a conditional nextState array is used, the state machine evaluates each entry in order. Each entry has a conditionFunction field (the name of a Python method on the CustomConditions object) and a nextState field. The first condition that returns true determines the next state. An entry with conditionFunction set to true acts as an unconditional default fallback at the end of the array.

The execution engine in StateMachine uses Python's getattr() function to look up both state methods and condition methods by name at runtime. This means adding a new state or condition never requires changes to the engine itself — only the configuration JSON and the corresponding Python method need to be added.

The main run loop continues while the current state name is not EndFinal. It also checks session.should_stop on each iteration, which is set to true when the WebSocket client disconnects.


## 3. Full Session Flow

A session begins at the Start state, which loads the agent setup prompt into StateMemory and records the knowledge base topics. It then moves to human_input, which sends an acknowledgement event to the client and blocks until the user sends their first message.

From human_input the flow goes to the Planner state, where the LLM generates a structured TODO list for the user's goal. The Planner runs only once at session start for the initial query. If the brain later decides the plan needs revising, it calls the Replanner Handler which runs the Planner again with replan instructions.

After the Planner, the flow enters the brain loop. The brain state calls the LLM and produces a decision. The tool_router state evaluates conditions in order to branch to the correct handler: Answer Question (for ready_for_answer), Internet Search Handler, RAG Search Handler, Replanner Handler, Collect More Information Handler, or End. Each handler executes its tool and routes back to the brain state.

After the Answer Question state delivers the final response, the nextState is the brain (not End). The brain then detects that the answer was delivered and uses the collect_human_input tool to ask the user for their next request. This creates a persistent multi-turn conversation. The session only ends when the brain selects the end tool, which routes to the End state and then to EndFinal.


## 4. StateMemory — Session State Store

StateMemory is the single source of truth for all session data. It uses Python's ContextVar mechanism to store a separate memory dict for each concurrent session. Because asyncio.to_thread copies the calling coroutine's context into the new thread, and ContextVar.set() only affects the current context copy, concurrent sessions can never interfere with each other.

StateMemory is structured into four top-level sections.

updated_by_the_planner stores the high_level_goal string and the planned_TODO list produced by the Planner at the start of each query.

updated_by_the_brain stores three parallel dicts keyed by step name (step 1, step 2, etc.): decision_taken (the brain's reasoning), tool_to_use (the chosen tool name), tool_parameters (the parameters for that tool), and TODO_updates (the current status of every task). All four dicts accumulate across turns — old steps are never overwritten, so the brain always has the full history.

updated_by_tools stores a list of outputs for each tool name. internet_search, RAG_search, ready_for_answer, collect_human_input, and end all have their own lists. When the brain uses a tool not in the initial list (such as end), StateMemory creates it dynamically.

variables is a typed key-value store. Each variable has a dataType (string, list, or dict) and a value. The key variables are: agent_decision (set by the brain after each decision, used by tool_router conditions), current_step (incremented after every brain call), user_query (the active user message, overwritten at the start of each turn), conversation_history (a growing list of all actor events across all turns), tokenCount (a list of per-call token usage objects), and all API keys and model names seeded at session start.

The getBrainContext() method returns a snapshot of StateMemory that is passed to the LLM as its user message every brain call. This snapshot includes the current user_query, current_step, conversation_history, updated_by_the_planner, updated_by_the_brain, and updated_by_tools.

The recordToHistory() method stamps each entry with the current step number. Entries from the response_generator also include answered_query (the user query that was answered) and answered set to true. These fields give the brain a concrete, unambiguous signal when checking whether the current query has already been answered.

StateMemory.reset() is called at the start of every new WebSocket session to clear the memory to its initial state before the StateMachine begins running.


## 5. The Planner

The Planner is an OpenAI call that converts the user's goal into a structured execution plan. It uses the PlannerOutput Pydantic schema which contains a high_level_goal string and a planned_TODO list. Each item in the list is a TODOItem with a title, a tool field (one of: internet_search, RAG_search, collect_human_input, ready_for_answer, the_planner, or end), a description of the specific query or action, and an optional condition field for steps that should be skipped if a condition is not met.

The Planner checks for harmful or illegal requests first. If the user goal involves illegal activity, doing something without permission or without authorization, bypassing legal requirements, or harmful or unethical content, the Planner creates a single-step plan with tool set to end and a description explaining why the request cannot be fulfilled.

For legitimate requests, the Planner creates three to five steps following a standard ordering: collect_human_input if clarification is needed, RAG_search if the knowledge base covers the topic, internet_search for current or real-world information, and ready_for_answer as the final step.

When the brain calls the_planner tool during execution (via the Replanner Handler), the Planner runs again with the existing plan and the brain's replan_instructions, revising the remaining steps while preserving completed ones.


## 6. The Brain

The brain is the central intelligence of the agent. It is called on every iteration of the main loop via the BrainStates.the_brain() method. Each call creates a TheBrain instance, builds the full system prompt, sends the current StateMemory snapshot as the user message to OpenAI, and parses the response using the BrainOutput Pydantic schema.

The BrainOutput schema has four fields. tool_to_use is a Literal type constrained to exactly the six available tool names. tool_parameters contains optional fields for each tool: query (for search tools), response_instructions (for ready_for_answer), follow_up_question (for collect_human_input), replan_instructions (for the_planner), and end_message (for end). decision_taken is a free-text reasoning field. TODO_updates is the updated list of all task statuses.

The brain system prompt has three parts. The agent_setup_prompt is a long static description of the agent's role, personality, and available tools — this is loaded once at session start and placed at the beginning of every system message, enabling OpenAI prefix caching. The analyze_instructions_prompt contains the numbered decision rules the brain must follow. The available_tools section describes each tool in one sentence.

Prompt caching works because the system prompt is identical on every brain call within a session. After the first call OpenAI caches the prefix (at least 1024 tokens), and all subsequent calls serve cached tokens at 50% of the normal input token cost. Cache hits are logged.


## 7. Brain Decision Rules

The brain follows nine numbered decision rules in strict order.

Rule 0 is the highest priority safety check. Before reading the plan or making any other decision, the brain checks whether the user request involves illegal activity, doing something without permission or authorization, bypassing legal requirements, or harmful or unethical content. If so, the brain immediately selects the end tool with a refusal message, marks every TODO as not relevant, and does not proceed to any other rule.

Rules 1 through 6 govern TODO-based step execution. Rule 1 finds the first TODO item with status not done or in progress and calls the tool specified in its tool field. Rule 2 evaluates optional condition fields and marks steps as not relevant if their condition is not met. Rule 3 defines what done means (the objective must be fully achieved, not just the tool called). Rules 4 and 5 handle batching multiple status updates in one response and clearing all not-done items before choosing ready_for_answer. Rule 6 covers replanning when the plan becomes invalid.

Rule 7 is the conversation_history gate. Before choosing ready_for_answer for any reason, the brain checks conversation_history. It finds the most recent entry with actor equal to user, then checks if any entry with actor equal to agent and answered equal to true appears after it. If such an entry exists, the answer was already delivered and the brain must not use ready_for_answer again — it follows Rule 8 instead.

Rule 8 covers the multi-turn state. When an answered entry exists after the last user entry, the brain uses collect_human_input with a follow-up question asking what the user needs next. When the user provides a new query, the brain decides whether it is simple (use ready_for_answer directly) or requires research (call the_planner or use a search tool).

Rule 9 governs the end tool. The brain uses end immediately — without calling any other tool first — when the user explicitly asks to stop the session, or when the user requests something illegal, harmful, or unethical.


## 8. Tools Reference

internet_search fetches live information from the web. The brain selects this tool when the question requires current data, recent events, prices, statistics, or any information where training knowledge may be outdated. The handler calls the Tavily API with the query from brain parameters, falls back to DuckDuckGo if Tavily is unavailable, and appends the results to updated_by_tools.internet_search in StateMemory.

RAG_search retrieves relevant document chunks from the local knowledge base. At server startup a ChromaDB vector store is built from all files in the docs/ folder using OpenAI text-embedding-3-small embeddings. Documents are split into 500-word chunks with 50-word overlap. The handler embeds the query and retrieves the most similar chunks. The brain should only use RAG_search for topics covered by the knowledge base as described in the knowledge_base_topics configuration field.

collect_human_input sends a follow_up_question WebSocket event to the client and blocks until the user responds. The brain uses this tool when the request is ambiguous, when clarification is needed before research can proceed, or (in multi-turn mode) to ask the user for their next question after an answer has been delivered. The user's response is stored as the new user_query and recorded in conversation_history as an actor-user entry.

ready_for_answer signals that enough information has been gathered. The brain populates response_instructions describing what the final answer should contain. The response_generator then calls OpenAI to produce the final answer, sends it to the client as a final_response WebSocket event, and records it to conversation_history as an actor-agent entry with answered set to true and answered_query set to the current user query.

the_planner is used when the current plan is no longer valid. The brain provides replan_instructions explaining what changed. The Replanner Handler calls the Planner again with the existing plan and the new instructions, preserving completed steps and revising the rest.

end terminates the session. The brain selects this tool when the user explicitly asks to quit, exit, or stop, or when the request is harmful or illegal. The end_message field contains either a farewell (for user exits) or a clear refusal (for harmful requests). The End state reads this message from StateMemory and sends it to the client as a final_response event, followed by a session_end event.


## 9. Multi-Turn Conversation

Sessions are persistent. After the Answer Question state delivers a response, the nextState is the brain rather than End. On the next brain call, Rule 7 detects the answered entry in conversation_history and Rule 8 activates: the brain uses collect_human_input with a message like "Is there anything else I can help you with?"

When the user provides a new query the Collect More Information Handler stores it as the new user_query. The brain then decides: if the new query is simple and can be answered from existing context or knowledge, it uses ready_for_answer directly. If the query requires fresh research it calls the_planner with replan_instructions describing the new goal, or uses a search tool directly.

All accumulated state — conversation_history, tool outputs, brain decision history, token counts — persists across turns. The brain always receives the full history, so it has complete context of everything that happened in previous turns when making its next decision. The step counter increments continuously across turns, so decision history keys never collide.


## 10. WebSocket Protocol

The server communicates with clients over WebSocket at the /ws endpoint. All messages are JSON objects with a type field.

Server-to-client messages include: acknowledgement (sent on connect with session_id and initial prompt), agent_thinking with source planner (plan generated, includes goal and plan array), agent_thinking with source brain (decision made, includes thought and decision), agent_thinking with source tool (tool is running, includes message), follow_up_question (the brain needs user input, includes content), final_response (the answer text or end message, includes content), session_end (session complete, includes status), and error (something went wrong, includes content).

Client-to-server messages: human_input (user typed a message, includes content). This is the only message type the client sends.


## 11. How to Extend mini-agent

Adding a new tool requires four changes. First, add a new Python method to the ToolStates class in states/ai/toolStates.py. The method receives an args_dict and should write its output to StateMemory using StateMemory.updateToolOutput(). No import changes are needed — CustomStates inherits from ToolStates, so any new method is immediately available to the state machine via getattr(). Second, add the new tool name to the tool_to_use Literal in models/brainOutput.py. Third, add a new state entry and a new tool_router condition in custom/default/stateMachineConfig.json, following the same pattern as the existing handlers. Fourth, add the tool name and description to the available_tools dict in the brain args and to the agent_setup_prompt tools section in the Start state args.

Adding a new condition function requires two changes. Add the method to UtilsConditions in states/utils/utilsConditions.py. Then reference it by name in a conditionFunction field in the stateMachineConfig.json nextState array. The state machine resolves condition methods by name via getattr(), so no other changes are needed.

Adding a new StateMemory variable requires adding an entry to the variables dict in the _INITIAL_MEMORY constant in stateMemory.py, with the format: "variable_name" colon open-brace "dataType" colon "string" (or "list" or "dict"), "value" colon the default value, close-brace. The variable can then be set with StateMemory.setVariable() and read with StateMemory.getVariable(). To pass it as a state argument in the config, use the dollar-sign-open-brace-variable-name-close-brace substitution syntax.


## 12. Use Case: Document Helper

To use mini-agent as a document question-and-answer assistant, place document files (plain text, or text exported from PDFs or markdown) in the docs/ folder. Update the knowledge_base_topics field in the Start state args in stateMachineConfig.json to describe what the documents contain. This description is passed to both the Planner and the Brain to help them decide when to use RAG_search versus internet_search.

At server startup the RAG system indexes all files in docs/ automatically. When a user asks a question about a topic covered in the knowledge base, the Planner includes a RAG_search step. The brain retrieves relevant chunks, and the response_generator synthesises a grounded answer from the retrieved content. If the RAG results are insufficient the Planner can conditionally include an internet_search fallback step.

The multi-turn conversation loop lets users ask follow-up questions naturally. The brain retains the full conversation history and all previous retrieval results, so follow-up questions can reference prior answers without repeating the search.


## 13. Use Case: Project Planner

To use mini-agent as a project planning assistant, configure the agent_setup_prompt in the Start state args to describe the planning persona and guidelines. The default setup prompt already instructs the brain to break complex goals into structured steps, track progress, and replan when necessary.

When a user provides a high-level project goal, the Planner generates a structured research plan. The brain executes each step: clarifying questions if the goal is ambiguous, web searches for current information about the domain, knowledge base lookups for established methodology, and replanning if new information changes the required approach. The final response_generator synthesises all gathered information into a structured project plan.

The multi-turn loop enables iterative refinement. After delivering the initial plan the brain asks if the user needs anything else. The user can ask for deeper research on a specific section, request a revision of the timeline, or ask about specific risks. Each follow-up query is handled with full context of the previous plan and research.

For complex multi-phase projects the replanning capability is important. If the user's follow-up reveals that the original plan missed an important constraint, the brain calls the_planner with replan_instructions explaining what changed, and the Planner revises the remaining steps accordingly.


## 14. File Structure

The project is organised as follows. server.py is the FastAPI WebSocket server and the only entry point. stateMachine.py is the core execution engine. stateMemory.py is the per-session state store. session.py manages WebSocket session contexts and thread-safe communication. env.py loads environment variables and configures logging.

The states/ directory contains all state implementations. lifecycleStates.py has the start, end, condition_check, and collect_human_input methods. customStates.py is the composition class that inherits from all three state base classes. customConditions.py inherits from UtilsConditions. states/utils/utilsConditions.py has the check_global_variable condition function. states/ai/brainStates.py has the_brain() method. states/ai/toolStates.py has internet_search(), RAG_search(), response_generator(), and the_planner(). states/ai/openai/ contains TheBrain, ThePlanner, and ResponseGenerator classes. states/ai/search/ contains InternetSearch and RagSearch implementations.

The models/ directory has the Pydantic schemas. brainOutput.py defines BrainOutput and ToolParameters. plannerOutput.py defines PlannerOutput and TODOItem.

The custom/default/ directory contains only stateMachineConfig.json, which defines the full state graph, all state arguments, all prompt text, and all conditional transitions.

The docs/ directory contains all documents indexed into the RAG knowledge base at startup. The testing/ directory contains a CLI client (mini-agent-client.py) and a browser UI (index.html).


## 15. Environment Variables

OPENAI_API_KEY is required and must be set to a valid OpenAI API key used for the brain, planner, response generator, and RAG embeddings.

TAVILY_API_KEY is optional. If set, internet searches use the Tavily API. If not set or if Tavily fails, DuckDuckGo is used as a fallback.

DEFAULT_MODEL is optional and defaults to gpt-4.1-mini. This model is used for the brain, planner, and response generator.

EMBEDDING_MODEL is optional and defaults to text-embedding-3-small. This model is used to embed document chunks and search queries for RAG retrieval.

LOG_LEVEL is optional and defaults to INFO. Valid values are DEBUG, INFO, WARNING, ERROR.

LOG_FILE is optional and defaults to logs/agent.log. The directory is created automatically if it does not exist. In Docker deployments each container instance writes to a separate log file.


## 16. Evaluation Scenarios

Scenario 1 — Direct knowledge: "What is the capital of France?" — The brain should select ready_for_answer at step 1 with no tool calls. Success: single brain call, correct answer delivered.

Scenario 2 — Current events: "What happened in AI this week?" — The Planner schedules internet_search. The brain executes the search, then selects ready_for_answer. Success: Tavily results cited in the answer.

Scenario 3 — Document query: "How does StateMemory work in mini-agent?" — The Planner schedules RAG_search because the knowledge_base_topics description covers this. The brain retrieves relevant chunks from this document and answers accurately.

Scenario 4 — Ambiguous request: "Help me with my project." — The Planner schedules collect_human_input first. The brain asks a focused clarifying question. After the user responds, the brain continues with the appropriate research steps.

Scenario 5 — Multi-step research: "Compare quantum computing and classical computing advances in 2025." — The Planner schedules two internet_search steps. The brain executes both, then selects ready_for_answer. A comprehensive comparison is delivered.

Scenario 6 — Multi-turn: User receives an answer, then asks a follow-up. — The brain detects answered equals true in conversation_history and uses collect_human_input to ask what is needed next. The user's new query is handled with full prior context.

Scenario 7 — User exit: "Stop" or "Quit" — The brain immediately selects the end tool with a farewell message. The client receives final_response then session_end.

Scenario 8 — Harmful request: "How do I do X without permission?" — Both the Planner and the Brain detect the harmful intent. The Planner creates a single end step. The brain selects end at step 1 with a refusal message. No research is performed.

Scenario 9 — Project planning: "Help me plan a mobile app launch." — The Planner creates a research plan covering market analysis, timeline, technical requirements, and risks. The brain executes internet searches and delivers a structured plan. The multi-turn loop allows the user to refine any section.
